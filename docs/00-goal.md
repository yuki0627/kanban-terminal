# kanban-terminal02 ゴールイメージ

このドキュメントは kanban-terminal02 の「行き着く先」を定義する。
実装・設計・優先順位の判断はすべてこの文書を基準にする。
ここに書かれていないこと、利用者と認識合わせできていないことを、推測で仕様化しない
(元祖 kanban-terminal `docs/00-overview.md` の開発方針を引き継ぐ)。

書き方も元祖の流儀に従い、**確認済み** と **未決事項** を明確に分ける。
未決事項は利用者との対話で決め、決まったら確認済みへ移す。

## 一言でいうと

**いくつものターミナルをカンバンボードで管理する、`npx` で起動できる軽量 Web アプリ。**

カード 1 枚 = ターミナル 1 つ。ターミナルで何かが走り出すとカードは In Progress へ、
終わると In Review へ自動で動く。AI エージェント(Claude Code 等)を走らせた場合は
さらに一段賢くなり、エージェントの作業中は In Progress、人間の入力を待った瞬間に
In Review へ動く。「どの作業が進行中で、どれが人間を待っているか」がボードを
一目見れば分かる、これが提供する価値のすべてである。

## 価値基準 — すべての判断の物差し

このプロダクトの価値は「動くこと」ではなく **UX** である(利用者が繰り返し
明示している最重要方針)。仕様・実装・レビューのあらゆる判断は、次の原則から
導出する。迷ったら効率・実装都合・元仕様の踏襲ではなくこの原則に従い、
原則から答えが導けない場合は利用者に確認する。

1. **開いた瞬間、続きがそこにある。** カードを開いたとき、作業の続きが
   完全な状態で目の前にあること。待たせない、失わせない。
2. **ひと手間を挟まない。** 目的(作業に戻る・状態を知る)までの操作を
   最小にする。「開いてからもう一操作」は敗北である。
3. **ボードを一目見れば分かる。** どの作業が進み、どれが人間を待っているか。
   レーン・色・通知はすべてこのためにある。
4. **先回りのリソース節約をしない。** 「メモリがもったいないから使われるまで
   起動しない」「アイドルになったら自動で殺す」のような節約設計は、
   その代金をユーザー体験(ひと手間・待ち時間・状態の喪失)で支払わせる。
   まず全部生かして体験を最優先し、節約を考えるのは、メモリ表示の数字が
   実際に問題を示したとき(例: カードを溜めたらマシンが重くなった)だけ。
   想像上のリソース不安で体験を削らない。

なぜ明文化するか: 実装者(AI を含む)は放っておくと「遅延起動」「省リソース」
「元仕様の踏襲」といった工学的デフォルトで仕様の空白を埋める。本プロジェクトでは
**UX 原則からの導出が常に優先**である(2026-07-06、ターミナル起動タイミングの
認識ズレを二度起こした反省から。経緯は `docs/09-open-questions.md` 論点8)。

## 成り立ち — なぜこの形か

この節は歴史的経緯であり、以後の判断の前提となる。

1. **元祖 `yuki0627/kanban-terminal`(Electron アプリ)** — ゴール
   (カンバン × AI エージェントのターミナル管理)と UI はしっかり文書化され、
   正しかった。しかし**ターミナル画面の永続化を自前実装した層**
   (scrollback の保存・再生、リサイズ reflow、ANSI 色復元…)がバグの温床となり、
   ターミナルとして使い物にならないレベルに達した。
   カンバン側のロジック起因のバグはほとんど無かった。
2. **mulmoterminal(receptron/mulmoterminal、MIT)** — 同じ問題を
   「自前実装ではなく tmux への委譲」で解いていた。実 PTY + tmux 永続化 +
   3 段階レジューム + hook 駆動の活動検知という安定したターミナル基盤を持ち、
   `npx` 一発で起動できることを実証している。
3. **kanban-terminal02(本リポジトリ)** — 元祖のゴールはそのまま、
   ターミナルの土台だけを mulmoterminal に交換したもの。

**過去の反省(このドキュメントが存在する理由):** 土台交換の際に
mulmoterminal 側へ寄せすぎ、看板ボードが「7 ビュー中の 1 タブ」に降格し、
mulmoterminal / MulmoClaude エコシステムの機能(帳簿・Collections・Wiki・PRs 等)まで
プロダクトに残ってしまった。ゴールイメージが文書として共有されていなかったことが
根本原因である。**看板ボードが主役であり、mulmoterminal は見えない土台である。**

## ゴールの姿(確認済み)

### プロダクトの輪郭

- アプリを開くと**カンバンボードが表示される**。看板ボードが唯一の主役画面である。
- mulmoterminal 由来の他ビュー(Chat / Grid / Files / Run)は**すべて削除**し、
  看板専用にする。Files / Run 相当は将来、カード展開ウィンドウ内の補助機能として
  必要になったら再検討する。
- `npx kanban-terminal` で起動できる軽量な Web アプリとして配布する
  (サーバー + ブラウザ UI。npm パッケージ名は `kanban-terminal`。
  元祖は npm 未公開のプロトタイプなので、製品版である本リポジトリがこの名を使う)。
- Electron デスクトップアプリの路線には戻らない。

### ボード UI — 元祖 kanban-terminal を踏襲する

UI のゴールイメージは元祖の `docs/04-ui-tui-spec.md` である。骨子:

- ステータスレーンは **To Do / In Progress / In Review / Done / Canceled** の 5 本。
- 左側に折りたたみ式の **Projects sidebar**:
  - 通常プロジェクトと **「Project なし」を同列のプロジェクト一覧**として扱う。
  - プロジェクト = 登録した作業フォルダ。追加はディレクトリ選択だけで完了し、
    表示名はフォルダ末尾名から自動生成する。
  - プロジェクトごとに**固有色**を割り当てる(Project なしも固有色を持つ)。
    プロジェクト行・カード・展開ウィンドウで同じ色を使い、紐づきを見せる。
  - プロジェクト行のクリックは「選択」ではなく**カード表示/非表示のトグル**。
    表示中プロジェクト 0 件も許容し、その場合も空レーンを表示する。
  - プロジェクト行内に、そのプロジェクト用の**カード追加 `+`** を置く。
- **Archive** はレーンとは別のカード属性。archived カードは通常ボードから消え、
  ボード右上の Archive 一覧から開く・復元できる。
  Archive まわりの UI は使い勝手を工夫すること(利用者の明示要望)。
- カードをクリックすると**展開ウィンドウ**が開き、中に **Memo と Terminal** を持つ。
  展開ウィンドウのサイズ・位置は保存し、次回開くときに復元する
  (利用者がドラッグで決めた値だけを保持する — 元祖仕様)。
- ボードの空白からの矩形選択で複数カードを選び、まとめて Archive できる。

### レイヤーモデル — カードはターミナルであり、AI はその上の層

本プロダクトの構造は 4 層で考える(下が基盤)。**一番の基本は「看板ボードの
カード 1 枚 1 枚がターミナルになっている」ことであり、AI エージェントは
その上で機能する追加の層**である(利用者の明示方針、2026-07-06)。

- **L1: ターミナル(基盤)** — カード = ただのターミナル。そのプロジェクトの
  cwd でシェルが開き、ターミナルでできることは何でもできる
  (`yarn install`、Web サーバー起動、vim、AI エージェント…)。
  カード作成時に「種別」は選ばない。
- **L2: プロセス検知(全カード共通)** — 前面で何かプロセスが走り始めたら
  **In Progress**、終わったら **In Review**(「終わったので結果を見て」の状態)。
  Web サーバーや `yarn dev` のような終わらないプロセスは、走っている間ずっと
  In Progress に居続ける(意図通り・確認済み)。
- **L3: AI エージェント検知(L2 に優先)** — エージェントは会話の
  キャッチボールをするため、プロセスが生きたまま「入力待ち」になる。
  hook により、エージェント作業中 → **In Progress**、人間の入力を求めた/
  停止した → **In Review**。エージェントが紐づいたターミナルでは L3 が
  L2 の判定に優先する。
- **L4: 会話の継続性(エージェントのみ)** — session ID・resume 情報を
  永続化し、tmux ごと消えた場合(マシン再起動等)でも `claude --resume` で
  会話を取り戻せる。エージェント以外のプロセスに resume は存在しない
  (tmux によるサスペンドが唯一の継続性)。

### カード — ターミナルを持つ「名前の付いた作業」

- **すべてのカードはターミナルを 1 つ持つ**。中身は**ただのシェル**であり、
  作成時に種別は選ばない(レイヤーモデル L1)。
- **カードは常に生きたターミナルを持つ。「未起動」という状態は存在しない**
  (利用者の明示方針、2026-07-06)。カードを作った瞬間にシェルが tmux 上で
  起動し、Archive で畳まれるまで生き続ける(論点1 の常時サスペンドと
  一体の不変条件)。**カードを開く = 生きているターミナルに繋ぐだけ**なので、
  ひと手間なく即座に使える。
  マシン再起動などでターミナルが死んでいた場合は、自動でシェルを起動し直して
  この不変条件を保つ(エージェントの会話の復元は L4 の resume)。
  シェル自体は軽い(1 個あたり数 MB)ので、使われないシェルがプロンプトの
  まま待っていても問題ない。
  ターミナルを一度も使わないままカードを完了・Archive してもよい
  (メモだけの TODO カードとしても機能する)。
- カードは**名前とメモ**を持ち、**閉じた状態でも何のターミナルかが分かる**。
  名前は手動が基本。エージェントのセッションが紐づいたら、その会話タイトルを
  未命名カードの自動初期値として使い、手動でいつでも上書きできる。
- カードはプロジェクト(または Project なし)に属し、ターミナルはその
  プロジェクトのフォルダを cwd として起動する。
  **Project なしのカードはホームディレクトリで起動する。**
- プロジェクト登録時、そのフォルダの**既存 Claude セッション(resume 候補)を
  一覧から選んでカードとして取り込める**(元祖の「セッション候補からのカード化」)。

### ターミナル — mulmoterminal の基盤をそのまま使う

- 実体はサーバーサイド PTY + tmux 永続化 + 3 段階レジューム
  (生きた PTY へ再接続 → tmux 再アタッチ → `claude --resume`)。
- **ターミナルの永続化・画面復元を自前実装しない。** これは元祖の失敗の核心であり、
  最重要の非ゴールである。
- フロントの xterm 接続管理(`useTerminalConnections` のシングルトン、
  attach/detach による DOM 再親化)もそのまま使う。

### ターミナルの寿命 — 常時サスペンド(確認済み)

判断基準は UX である。**「カードを開いた瞬間、作業の続きがそこにあること」**が
看板ターミナルの存在意義であり、寿命ポリシーはリソース効率ではなくこの体験で決める。

- **デフォルトは常時サスペンド。** カード(展開ウィンドウ)を閉じても
  ターミナルは tmux 上で生き続け、再度開けば画面・スクロールバック・
  実行中のプロセスすべてが即座に完全復元される。
  レーン移動(Done / Canceled を含む)でも自動終了しない。
- **終了は人間の明示操作のみ。**
  - **Archive がターミナルを畳む唯一の解放ポイント**
    (「この作業は畳んだ」という明示宣言)。実行中のプロセスがある場合は
    確認ダイアログを出す。
  - カード上の明示的な終了操作でも閉じられる。
- **メモリ使用量を可視化する。** tmux のペインからプロセスツリーの実メモリを
  合算し、カード単位のメモリ使用量とボード全体の合計を指標として表示する
  (当初は控えめな表示でよいが、常に確認できる場所に置く)。
  「全部生かす」の代償を推測でなく数字で監視し、ポリシー変更が必要かは
  観測にもとづいて判断する。
- **マシン再起動などで tmux ごと消えた場合はレジュームで復帰する**
  (`claude --resume` で会話は戻る。画面と実行中プロセスは失う)。
  この保険として、session ID・resume 情報などの**参照の永続化**を維持する
  (元祖が cmux から採用した「継続性 = ライブプロセスの延命ではなく、
  参照を永続化して辿れること」という思想を引き継ぐ)。
  エージェント以外のプロセスにはレジュームが存在しないため、Archive 時の
  確認ダイアログが実質の保護になる。

### レーン自動移動 — レイヤーモデルに従う

レーンを動かすシグナルは 2 系統あり、**L3 が L2 に優先する**。

**L2: プロセス検知(全カード共通)** — tmux が教えるペインの前面プロセスを
サーバーが監視し、シェル以外のプロセスが前面に来たら「作業開始」、
シェルに戻ったら「終了」とみなす:

- プロセス開始 → **In Progress**
- プロセス終了 → **In Review**
- 終わらないプロセス(Web サーバー等)は In Progress に居続ける(確認済み)

**L3: エージェント hook(エージェントが紐づいたターミナルで優先)**:

- `working`(UserPromptSubmit)→ **In Progress**
- `waiting`(Stop / Notification)→ **In Review**

**共通の不変条件**(元祖で確立、`kanbanBoard.ts` に実装済みのものを維持):

- `idle` はシグナル不在であり、**行き先ではない**(サーバー再起動で全員 idle に
  なってもカードは現在のレーンに留まる)。
- 遷移は状態変化時のみ発火(エッジトリガー)。
- To Do / Done / Canceled は人間がドラッグで決める。手動で完了させたカードは
  終了・停止系のシグナルでは引き戻されず、**実作業開始(L3 の working、
  または L2 のプロセス開始)のみ**が In Progress へ再オープンする。
- 閉じたカードで自動移動が起きたら unread マークを付け、開いたら解除する。
- 何も走っていない(前面がシェルのままの)カードはシグナル不在であり
  自動移動しない(手動ドラッグのみで動く)。

**L3 の hook の刺し方:** カードのターミナルはただのシェルなので、利用者が
シェル内で打った `claude` にも hook が効く仕組みが必要になる。
第一候補は **PATH シム方式**(カードのシェルの PATH 先頭にシムを置き、
`claude` 起動時に session ID の指定と hook 設定の注入を自動で行う。
利用者のグローバル設定は汚さない)。代替は元祖方式(グローバル hook 設定の
マージ + 環境変数によるカード紐づけ)。最終決定は `docs/01-domain-model.md` で行う。

**対応エージェントの方針(確認済み):** 最初のリリースは Claude Code のみだが、
**エージェント種別(agent kind)を追加できる仕組みとして設計する**。
リリース直後に Codex を追加し、以後も利用者の要望に応じて追加していく。
hook の刺し方・活動状態への変換・resume コマンドの組み立てをエージェント種別ごとに
差し替えられる抽象を最初から置く(元祖の `KANBAN_TERMINAL_AGENT_KIND` の発想)。

### 永続化

- ボード状態(プロジェクト・カード・レーン・名前・メモ・Archive)は
  **サーバー側で永続化**する(元祖の `~/.kanban-terminal/board.json` に相当)。
  ブラウザの localStorage には置かない(現状の localStorage は暫定実装)。

### MVP(最初のリリース)の範囲

Projects sidebar、カードの名前/メモ、hook 駆動の自動レーン移動に加えて、
以下も**すべて MVP に含める**(確認済み):

- Archive(UI は使い勝手を工夫する)
- プロジェクト固有色
- 展開ウィンドウのサイズ・位置保存
- 複数選択・一括 Archive
- メモリ使用量の可視化(カード単位+ボード全体の合計)

## 非ゴール(確認済み)

以下は kanban-terminal02 には**含めない**。フォーク元から継承したものは削除する。

- **帳簿・会計機能**(Accounting / MulmoBooks)— MulmoTerminal / MulmoClaude 側の機能
- Collections / Feeds ビュー
- Wiki ビュー
- PRs & Issues ビュー
- Chat / Grid / Files / Run ビュー(mulmoterminal の画面構成)
- GUI プラグイン基盤(GUI MCP / plugins-registry / PluginFrame / GuiPanel)
- MulmoClaude 共有ワークスペース(`~/mulmoclaude`)前提のバックエンド群
- Electron デスクトップアプリ化
- ターミナル永続化・画面復元の自前実装

残っている論点は `docs/09-open-questions.md` で一つずつ議論して決める
(決まったものから本文書または後続ドキュメントへ反映する)。主なもの:

1. **Archive UI の具体案** — 「工夫してほしい」の中身。UI 案を複数出して
   利用者が選ぶ(元祖の `docs/ui-proposals/` 方式を踏襲する)。
2. **エージェント抽象の具体設計** — hook の刺し方・状態変換・resume の

3. カードとセッションの結びつきの定義、通知(音・favicon)の扱い、
   サーバーの単一インスタンス方針、upstream 追従方針
   — 詳細は `docs/09-open-questions.md` を参照。
