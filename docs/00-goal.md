# kanban-terminal02 ゴールイメージ

このドキュメントは kanban-terminal02 の「行き着く先」を定義する。
実装・設計・優先順位の判断はすべてこの文書を基準にする。
ここに書かれていないこと、利用者と認識合わせできていないことを、推測で仕様化しない
(元祖 kanban-terminal `docs/00-overview.md` の開発方針を引き継ぐ)。

書き方も元祖の流儀に従い、**確認済み** と **未決事項** を明確に分ける。
未決事項は利用者との対話で決め、決まったら確認済みへ移す。

## 一言でいうと

**AI エージェント(Claude Code 等)のターミナルセッションをカンバンボードで管理する、
`npx` で起動できる軽量 Web アプリ。**

カードはターミナルであり、エージェントの活動状態(作業中 / 入力待ち)に応じて
カードが自動でレーンを移動する。「どの作業がどこまで進み、どれが人間を待っているか」が
ボードを一目見れば分かる、これが提供する価値のすべてである。

## 成り立ち — なぜこの形か

この節は歴史的経緯であり、以後の判断の前提となる。

1. **元祖 `yuki0627/kanban-terminal`(Electron アプリ)** — ゴール
   (カンバン × AI エージェントのターミナル管理)と UI はしっかり文書化され、
   正しかった。しかし**ターミナル画面の永続化を自前実装した層**
   (scrollback の保存・再生、リサイズ reflow、ANSI 色復元…)がバグの温床となり、
   ターミナルとして使い物にならないレベルに達した。
   カンバン側のロジック起因のバグはほとんど無かった。
2. **mulmoterminal(receptron/mulmoterminal、MIT)** — 同じ問題を
   「自前実装ではなく tmux への委譲」で解いていた。実 PTY + tmux 永続化 +
   3 段階レジューム + hook 駆動の活動検知という安定したターミナル基盤を持ち、
   `npx` 一発で起動できることを実証している。
3. **kanban-terminal02(本リポジトリ)** — 元祖のゴールはそのまま、
   ターミナルの土台だけを mulmoterminal に交換したもの。

**今回の反省(このドキュメントが存在する理由):** 土台交換の際に
mulmoterminal 側へ寄せすぎ、看板ボードが「7 ビュー中の 1 タブ」に降格し、
mulmoterminal / MulmoClaude エコシステムの機能(帳簿・Collections・Wiki・PRs 等)まで
プロダクトに残ってしまった。ゴールイメージが文書として共有されていなかったことが
根本原因である。**看板ボードが主役であり、mulmoterminal は見えない土台である。**

## ゴールの姿(確認済み)

### プロダクトの輪郭

- アプリを開くと**カンバンボードが表示される**。看板ボードが唯一の主役画面である。
- `npx` で起動できる軽量な Web アプリとして配布する
  (mulmoterminal の `npx mulmoterminal` 相当。サーバー + ブラウザ UI)。
- Electron デスクトップアプリの路線には戻らない。

### ボード UI — 元祖 kanban-terminal を踏襲する

UI のゴールイメージは元祖の `docs/04-ui-tui-spec.md` である。骨子:

- ステータスレーンは **To Do / In Progress / In Review / Done / Canceled** の 5 本。
- 左側に折りたたみ式の **Projects sidebar**:
  - 通常プロジェクトと **「Project なし」を同列のプロジェクト一覧**として扱う。
  - プロジェクト = 登録した作業フォルダ。追加はディレクトリ選択だけで完了し、
    表示名はフォルダ末尾名から自動生成する。
  - プロジェクトごとに**固有色**を割り当てる(Project なしも固有色を持つ)。
    プロジェクト行・カード・展開ウィンドウで同じ色を使い、紐づきを見せる。
  - プロジェクト行のクリックは「選択」ではなく**カード表示/非表示のトグル**。
    表示中プロジェクト 0 件も許容し、その場合も空レーンを表示する。
  - プロジェクト行内に、そのプロジェクト用の**カード追加 `+`** を置く。
- **Archive** はレーンとは別のカード属性。archived カードは通常ボードから消え、
  ボード右上の Archive 一覧から開く・復元できる。
- カードをクリックすると**展開ウィンドウ**が開き、中に **Memo と Terminal** を持つ。

### カード — ターミナルであり、識別できる

- **カード = 1 つのターミナル(セッション)**。カードを開けばそのターミナルに戻れる。
- ただしカードは**名前とメモ**を持ち、**閉じた状態でも何のターミナルかが分かる**。
  ターミナルの使い捨てウィンドウではなく、「名前の付いた作業」としてボードに残る。
- カードはプロジェクト(またはProject なし)に属し、
  ターミナルはそのプロジェクトのフォルダを cwd として起動する。

### ターミナル — mulmoterminal の基盤をそのまま使う

- 実体はサーバーサイド PTY + tmux 永続化 + 3 段階レジューム
  (生きた PTY へ再接続 → tmux 再アタッチ → `claude --resume`)。
- **ターミナルの永続化・画面復元を自前実装しない。** これは元祖の失敗の核心であり、
  最重要の非ゴールである。
- フロントの xterm 接続管理(`useTerminalConnections` のシングルトン、
  attach/detach による DOM 再親化)もそのまま使う。

### 自動レーン移動 — hook 駆動(実装・検証済み)

Claude Code の hook をセッションに注入し、活動状態でカードを動かす。
ルールは元祖で確立し、現行の `kanbanBoard.ts` に移植済みのものを維持する:

- `working`(UserPromptSubmit)→ **In Progress**
- `waiting`(Stop / Notification)→ **In Review**
- `idle` はシグナル不在であり、**行き先ではない**(サーバー再起動で全員 idle に
  なってもカードは現在のレーンに留まる)。
- 遷移は状態変化時のみ発火(エッジトリガー)。
- To Do / Done / Canceled は人間がドラッグで決める。手動で完了させたカードは
  Stop / Notification では引き戻されず、実作業開始(working)のみが再オープンする。
- 閉じたカードで自動移動が起きたら unread マークを付け、開いたら解除する。

### 永続化

- ボード状態(プロジェクト・カード・レーン・名前・メモ・Archive)は
  **サーバー側で永続化**する(元祖の `~/.kanban-terminal/board.json` に相当)。
  ブラウザの localStorage には置かない(現状の localStorage は暫定実装)。

## 非ゴール(確認済み)

以下は kanban-terminal02 には**含めない**。フォーク元から継承したものは削除する。

- **帳簿・会計機能**(Accounting / MulmoBooks)— MulmoTerminal / MulmoClaude 側の機能
- Collections / Feeds ビュー
- Wiki ビュー
- PRs & Issues ビュー
- GUI プラグイン基盤(GUI MCP / plugins-registry / PluginFrame / GuiPanel)
- MulmoClaude 共有ワークスペース(`~/mulmoclaude`)前提のバックエンド群
- Electron デスクトップアプリ化
- ターミナル永続化・画面復元の自前実装

## 未決事項(利用者と対話して決める)

1. **mulmoterminal 由来の他ビューの扱い** — Chat / Grid(複数ターミナル格子)/
   Files(ファイルエクスプローラ)/ Run(スクリプト実行)は残すか。
   案: すべて削除して看板専用にする。ただし Files と Run は展開ウィンドウ内の
   補助機能として将来検討の余地あり。
2. **カードの名前の付き方** — 手動入力のみか、セッションタイトル
   (Claude の会話タイトル)から自動で初期値を与えて手動で上書きか。
3. **「Project なし」カードの cwd** — フォルダに属さないカードのターミナルは
   どこで起動するか(ホームディレクトリ等)。
4. **元祖 UI 機能の段階分け** — Archive、Memo/Terminal 分割、プロジェクト色、
   複数選択一括 Archive、Hidden projects などをどの順で入れるか
   (ゴールには全部含む前提で、実装フェーズの優先順位の話)。
5. **既存セッションのカード化** — プロジェクト登録時に、そのフォルダの既存
   Claude セッション(resume 候補)をカードとして取り込むか。
6. **配布名** — npm パッケージ名・コマンド名(`kanban-terminal` は元祖と衝突する。
   `kanban-terminal02` のままか、別名か)。
7. **対応エージェント** — 当面 Claude Code のみか、元祖のように Codex も
   視野に入れるか(hook 連携の抽象化に影響)。
