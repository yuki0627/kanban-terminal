archive-ui-proposals.html で決めた「アーカイブ一覧をボード右端の折りたたみ帯に置く」という配置はそのまま前提にした続編。 今回のお題は配置ではなく操作: (1) アーカイブ済みカードをクリックしても現状は無反応 (通常カードは @click="openCard" でターミナルオーバーレイが開くが、 archived-card にはハンドラが無い)、(2) カードが増え続けて消えないことへの不安感 — の2点を解く4案。すべて「元に戻す」に加えて完全に削除という現状未実装の出口を新設する前提で設計した。 最初のラウンド(旧6番: 容量メーター+整理モード案)は「上限の根拠が分かりにくい」というフィードバックで見送り、 代わりによりシンプルな2案(6・7)を追加している。並び順(4〜)は優劣ではなく比較のための番号。

4. 詳細ポップオーバー — クリックで復元・完全削除

アーカイブ済みカードのクリックに初めて意味を持たせる案。通常カードの全画面オーバーレイ(ターミナル+メモ編集)は セッションが既に解放済みのアーカイブには重すぎるため、代わりにカードの隣に小さく浮かぶ「セッション終了レシート」を出す。 プロンプト行・monospace のメタ情報など、ターミナルアプリ自身の語彙でカードの最期を見せ、 その場で 元に戻す か 完全に削除 するかを決められる。あわせて、90日以上前のカードには埃(ダスト)色の 経過タグを付け、一覧の先頭に控えめな案内を出すことで「気づいたら消せる」状態を作り、溜まっていく不安を和らげる。

強み: アーカイブ済みカードのクリックが初めて機能する・Restore/完全削除がその場で1手で完結・古いカードが視覚的に目立つので「気づいたら消せる」・ターミナルの語彙をそのまま流用でき世界観が壊れない 弱み: ポップオーバーの表示位置がクリックしたカードの位置に依存し実装がやや複雑・一覧全体を見渡す検索/フィルタ用途には不向き（一覧性が要る時は案1・2と併用が現実的）

6. 既存オーバーレイの流用 — 新しい部品を増やさない案

アーカイブ済みカードをクリックしたら、通常カードを開くときと全く同じ overlay をそのまま開く。違うのはターミナル欄の中身だけで、セッションは既に解放済みなので「このカードはアーカイブ済みです（セッション終了）」という一言に置き換え、メモは読み取り専用にする。ヘッダーのボタンも既存の .overlay-close を使い回したまま、元に戻す・完全に削除・閉じるの3つに絞るだけ。新しいポップオーバーも経年フェードのような新しい視覚言語も足さない。完全に削除はブラウザ標準の確認ダイアログ(confirm())を一段階挟むだけで、専用の確認UIは作らない。溜まっていく不安への対処も最小限にとどめ、一覧の最下部に飾り気のないテキストリンク「アーカイブ済みを全て削除」を1つ置くだけで、件数や割合は出さない。

強み: 新しいUI部品をほぼ作らずに済み実装差分が最小・「クリックしたら開く」という通常カードと全く同じメンタルモデルをそのまま使い回せる・確認は標準 confirm() 一回だけで実装/学習コストが低い・見た目の一貫性が保たれ世界観が壊れない 弱み: 一覧全体を見渡しながら複数選択して一括操作する動線がない(基本は1件ずつ開いて操作)・セッションが無いのに全画面級のオーバーレイを開く体験がやや大げさに見える可能性・古いカードを目立たせる工夫が無く「気づいたら消せる」きっかけは案4・5より弱い

7. 操作はふたつだけ — クリックで復元、×で完全に削除

詳細ポップオーバー(4)でもインライン操作(5)でもなく、そもそも選べる操作の数を2つに減らすことでシンプルさを作る案。 アーカイブ一覧を開く理由はほぼ「これを元に戻したい」なので、カード本体をクリックしたら確認なしで即座に元のレーンへ復元する。 誤クリックの不安は確認ダイアログではなく、画面下に数秒だけ現れる取り消しトーストで担保する。 もう一方の操作である完全削除は、カード隅の×だけに切り出し、押すとブラウザ標準の confirm() をそのまま使う — アプリの外側からやってくる、装飾ゼロの一段階の壁として機能する。 ホバーで何かが現れたり色が変わったりする演出は使わない。一覧のどのカードも常に同じ見た目・常に同じ2つの操作しか持たない、 という一貫性そのものがこの案の価値。

強み: 覚えることが常に2つだけ(クリック=復元・×=完全削除)・クリック無反応バグを最も直接的に解消・「確認」ではなく「取り消せる」ことで安心感を担保するので心理的コストが低い・完全削除はブラウザ標準 confirm() に丸投げでき実装コストも最小 弱み: 一覧性より即時実行を優先するため「詳細を見てから決めたい」用途には向かない・confirm() は見た目を制御できずアプリのトーンから浮く(意図的な違和感だが好みが分かれる)・トーストの表示中にページを離れると取り消せない
