## 決定済み

### 論点1: ターミナルの寿命(2026-07-06 決定)

**問い:** カードが完了・Archive されたとき、裏で生きている tmux セッション
(とその中の claude / シェル)をどうするか。

**決定:**

- デフォルトは**常時サスペンド** — カードを閉じてもレーンが動いても自動終了しない。
  開けば即座に完全復元(画面・スクロールバック・実行中プロセス)。
- 終了は**人間の明示操作のみ**。Archive がターミナルを畳む唯一の解放ポイント
  (実行中プロセスがあれば確認ダイアログ)。カード上の明示終了操作も可。
- **メモリ使用量を可視化**(カード単位+ボード全体合計)。
  「全部生かす」の代償は推測でなく観測で監視し、ポリシー変更の要否は数字で判断する。
- tmux ごと消えた場合(マシン再起動等)は `claude --resume` によるレジュームで復帰。
  session ID・resume 情報など**参照の永続化**を保険として維持する。

**理由:**

- 判断基準は UX — 「開いた瞬間に続きがそこにある」ことが看板ターミナルの
  存在意義であり、動けばいいではなくユーザー体験を最優先する(利用者の明示方針)。
- ターミナル・tmux の層は実質タダ(アイドルシェルは数MB)。重いのは中身の
  claude(Node プロセス、アイドルでも数百MB級)だが、OS のメモリ圧縮もあり
  実感負荷は数字より軽い。成立するかは可視化した実測で確認する。
- 元祖が cmux 比較(元祖 `docs/13-cmux-comparison-decision-brief.md`)で出した
  「継続性 = ライブプロセスの延命ではなく参照の永続化」という思想を、
  レジューム側の保険として引き継ぐ。

**反映先:** `docs/00-goal.md`「ターミナルの寿命 — 常時サスペンド」節、
MVP 範囲(メモリ可視化)。

### 論点2: カードとセッションの結びつきの定義(2026-07-07 決定)

**問い:** カードのターミナルはただのシェル(論点7)になったため、利用者が
シェル内で打った `claude` / `codex` をどう検知し、レーン自動移動(L3)へ
どう結びつけるか。

**決定:** **フックは使わない(フックレス化)。** レーン自動移動の判定は
Claude hook + PATH シムではなく、サーバーが既に中継している PTY のバイト流
(入力・出力)で行う。

- 前面プロセスで 3 分岐: シェル → シグナル不在 / 既知エージェント
  (claude・codex) → PTY 活動判定 / その他プロセス → L2 既存どおり。
- PTY 活動判定: ユーザーの Enter 入力 → 1.5 秒以内に持続出力(3 チャンク
  以上)が始まったら working(In Progress)。working 中に出力が 2.0 秒
  より長く止まったら In Review(完了・承認待ちを区別しない — 利用者決定:
  どちらもユーザーのボールであれば十分)。Enter が先行しない出力(起動
  バースト・resume 再描画・アイドル中の孤立再描画)は working にしない。
- エージェント検出は `#{pane_current_command}` の単純一致では不十分
  (npm 版 claude は `node` になり得る)。ペイン PID からプロセスツリーを
  辿ってコマンドラインをマッチさせる。
- session ID の紐づけ(用途は L4 resume とカード自動命名のみ。レーン移動には
  不要)は、cwd スラグ配下で新規作成された transcript ファイルを起動時刻の
  相関で発見する方式(transcript 発見方式)。同一 cwd で複数カードが同時に
  エージェントを起動した場合の曖昧さは既知の制約とし、過剰な tie-break
  実装はしない。カード自動命名はセッション jsonl の `ai-title` 行を使う。

**理由:**

- 「気軽に使いたいときにフックの設定をするのはハードルが高い」(利用者、
  2026-07-07)。PATH シムはカードのシェル起動のたびに PATH を加工する
  仕組みであり、利用者体験を損なわないとしても「刺さる仕組みそのもの」の
  複雑さ・失敗モードが増える。
- 実測(2026-07-07、tmux + pipe-pane で claude(haiku)の PTY 出力を記録)。
  無音ツール実行(`sleep 20`)・WebSearch とも出力ギャップは最大 0.55 秒。
  2.0 秒の静止判定に対して約 3.6 倍のマージンがある。完了後のアイドルは
  孤立した数バイトの再描画が数秒おきに 1 回あるのみで、作業中と明確に
  区別できる。起動直後は約 1 秒のバースト描画のみで持続出力にならず、
  「Enter 非先行は working にしない」ルールで自然に除外される。
- ボールの二値(ユーザー / AI)だけで十分というのは利用者の明示判断で、
  #174 で分けた blocked/done の区別は本判定では不要(発行側で統合する)。

**反映先:** `docs/00-goal.md`「レーン自動移動」節・レイヤーモデル L3/L4、
`plans/feat-kanban-first-rebuild.md` Phase 3、
`plans/feat-hookless-lane-signals.md`(Codex 向け作業指示書)。

### 論点4: エージェント抽象の設計(2026-07-07 決定)

**問い:** Claude Code と Codex とで hook の刺し方が異なる(Claude は
per-spawn `--settings` 注入、Codex は同方式が使えない)。「hook の刺し方
そのもの」を差し替えられる抽象が必要か。

**決定:** 論点2 のフックレス化により、この技術リスクは消滅する。
PTY 活動監視はエージェント種別を問わず同じ仕組みがそのまま効く
(プロセス検出パターンだけがエージェント種別ごとに違う)。
AgentKind の継ぎ目は次の 3 点だけに縮小する薄いインターフェースでよい:

1. プロセス検出パターン(コマンドラインのマッチ条件)
2. resume コマンドの組み立て
3. カードタイトルの自動取得元(transcript の該当フィールド)

Codex は rollout ファイル(`~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`)
にターン完了時 `event_msg / task_complete` を明示的に書く(実測済み)。
将来これを PTY 判定のバックアップシグナルとして使う余地があるが、
今回のスコープ外(PTY 監視のみで判定を完結させる)。

**理由:** hook 注入という「刺し方」の差異こそが最大の技術リスクだった。
フックを使わない判定方式にした時点で、差し替えるべき対象がそもそも
存在しなくなった。

**反映先:** `docs/00-goal.md`「対応エージェントの方針」節、
`plans/feat-kanban-first-rebuild.md` Phase 6(AgentKind の継ぎ目)。

### 論点7: カードとターミナルのレイヤーモデル(2026-07-06 決定)

**問い:** カードは「AI エージェントのセッション」か「ただのターミナル」か。
カード作成時に種別(Claude / プレーンシェル)を選ぶのか。

**決定:** カード = ただのターミナル。種別選択は廃止する。構造は 4 層の
レイヤーモデルで考える(詳細は `docs/00-goal.md`「レイヤーモデル」節):

- L1: ターミナル(基盤)— カードはただのシェル。何でも動く。
- L2: プロセス検知(全カード共通)— 開始 → In Progress、終了 → In Review。
  終わらないプロセス(Web サーバー等)は In Progress のまま(意図通り・確認済み)。
- L3: エージェント検知(L2 に優先)— PTY 活動(フックなし)で作業中 /
  入力待ちを判定(論点2、2026-07-07 改定)。
- L4: 会話の継続性(エージェントのみ)— resume。

**理由:** PR #3 の途中実装を見た利用者の方向修正。「まず一番の基本は
看板ボードのカード 1 枚 1 枚がターミナルになっていること。AI エージェントは
その上で機能する層」(利用者)。

**反映先:** `docs/00-goal.md`(一言でいうと / レイヤーモデル / カード /
レーン自動移動)、`plans/feat-kanban-first-rebuild.md` Phase 3。

### 論点8: ターミナルの起動タイミング(2026-07-06 決定・同日改定)

**問い:** ターミナルはいつ起動するか。

**経緯と決定:** 当初は元祖踏襲の「ターミナル領域の操作時に起動」→ 利用者指摘で
「カードを開いたら自動起動」→ 確認の結果、さらに強いモデルに確定:
**カード = 常に生きたターミナル。「未起動」という状態は存在しない。**
カード作成と同時にシェルが tmux 上で起動し、Archive まで生き続ける。
開く = 生きているターミナルに繋ぐだけ(即座・ひと手間なし)。
マシン再起動等で死んでいたら自動でシェルを起動し直して不変条件を保つ
(会話の復元は L4 の resume)。

**反省(なぜ二度ズレたか):** 実装側が仕様の空白を「遅延起動・省リソース・
元仕様の踏襲」という工学的デフォルトで埋めたため。本プロジェクトの判断の
物差しは UX であり、論点1 で既に「開いた瞬間、続きがそこにある」と決めていた
以上、そこから導出すれば「常に生きている」に到達できたはずだった。
この反省を `docs/00-goal.md` の「価値基準」節(UX 4 原則)として恒久化し、
以後のすべての仕様判断はそこから導出する。

**反映先:** `docs/00-goal.md` 価値基準節(新設)・カード節・レーン自動移動節、
`plans/feat-kanban-first-rebuild.md` 大原則・Phase 3。

### Archive UI の具体案(2026-07-07 決定)

**問い:** ゴール文書の「Archive まわりの UI は使い勝手を工夫すること」
(利用者の明示要望)の中身をどうするか。

**決定:** `docs/ui-proposals/archive-ui-proposals.html` の 3 案
(①右上トレイ + ドロワー / ②専用オーバーレイ + 検索 / ③レーン末尾の
折りたたみ帯)から**案3「レーン末尾の折りたたみ帯」**を採用。
5 レーンの右端に Archive を 6 本目のレーンのような帯として常設する。
既定は折りたたみ(薄い縦帯 + 件数)、クリックで展開するとグレースケール調の
archived カードが並び、復元は帯からレーンへのドラッグで盤面上で完結する。

**理由:** モーダルを開かないぶん「ボードは一枚」という一目性を最も壊さない
(価値基準3「ボードを一目見れば分かる」からの導出)。既知の弱み(展開時に
レーンが狭くなる・検索/フィルタの置き場が別途必要)のうち、検索/フィルタは
最初は作らず、archived カードの件数が実際に増えて困ってから足す。

**反映先:** `docs/00-goal.md` ボード UI 節・未決事項、
`plans/feat-kanban-first-rebuild.md` Phase 5。

### 論点3: 通知(attention sound / favicon)の keep / cut

mulmoterminal の入力待ち通知(音・favicon バッジ)がゴール文書の keep にも
cut にも入っていない。看板の中心価値は「入力待ちに気づけること」なので
keep が有力(推奨: keep)。

### 論点5: サーバーの単一インスタンス方針

元祖は single-instance Electron。02 は Web サーバーなので `npx kanban-terminal`
を2回打ったときの挙動を決める(ボードは `~/.kanban-terminal` に1つなので、
1ユーザー1サーバー・既起動検出で既存を開く、が有力)。

### 論点6: upstream(mulmoterminal)追従方針

大量削除後は `git merge upstream/main` は事実上不可能。「ターミナル核の
修正だけ cherry-pick」への切り替えを `docs/03-decisions.md` に宣言する。
