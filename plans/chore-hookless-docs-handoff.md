## 決定事項（2026-07-07 Yuki 決定 — ここが本プランの根拠）

1. **フック・シムは使わない（フックレス化）。** 気軽に使いたいときにフック設定を
   要求するのはハードルが高い、という利用者判断。レーン自動移動のエージェント検知
   （L3）は Claude hook + PATH シムではなく、**サーバーが既に中継している PTY の
   バイト流（入力・出力）で判定する**。
2. **ボールの二値だけ分かればよい。** 「承認待ち(blocked)」と「完了(done)」の区別は
   不要。AI の返答が終わっていればユーザーボール（→ In Review）。#174 で分けた
   blocked/done は発行側で統合する（`kanbanBoard.ts` の遷移ルールは keep・無変更。
   発行側が常に done 相当を出すだけ）。
3. **PR #3 の進行:** Codex がまず PR #3 にフックレス差し替えコミット
   （シム撤去 + PTY 活動監視）を積んでからマージ（main に「消す予定のシム」を
   入れない）。Phase 5-6 は次 PR。
4. **Archive UI（Phase 5 で唯一の「利用者がモックから選ぶ」工程）は
   ハンドオフ前に潰す。** Claude（設計担当セッション）がモック 2〜3 案を先に作り、
   Yuki が選定し、決定をプランに書き込んでから Codex に渡す。
   これにより Codex は一度も止まらず最後まで走れる。

## 実測結果（2026-07-07、tmux + pipe-pane で claude(haiku) の PTY 出力を全記録）

生ログはセッション固有 scratchpad にあり消える。数値はここが正本。

| フェーズ | 実測値 | 含意 |
| --- | --- | --- |
| 作業中: `sleep 20`（無音 25s） | 258 チャンク、最大ギャップ **0.55s** | ツール実行中もスピナーは途切れない |
| 作業中: WebSearch（天気 ~11s） | 110 チャンク、最大ギャップ **0.52s** | ネットワーク待ちでも止まらない |
| 完了後アイドル（15s / 25s 区間） | **80B の孤立再描画が数秒おきに 1 回**（数字更新） | 静止はほぼ無音。作業中と明確に区別可 |
| 起動直後 | 約 1s のバースト描画 → 完全静止 | 起動は「持続出力」にならない |

また Codex CLI はセッションを `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` に
追記し、ターン完了時に `event_msg / task_complete` を明示的に書く（実測済み）。
将来 Codex 対応時のバックアップシグナルに使える（今回のスコープ外）。

## 確定した判定仕様（この定数をプラン・docs に書き込む）

前面プロセスで 3 分岐（分岐判定は下記「エージェント検出」参照）:

```text
シェル(zsh等)                  → シグナル不在（自動移動しない。L2 既存どおり）
既知エージェント(claude/codex) → PTY 活動判定（下記）
その他プロセス(ビルド/dev鯖等)  → L2 既存どおり（開始→In Progress、シェルに戻る→In Review）
```

エージェント枝の PTY 活動判定:

1. **working 開始** = エージェントが前面のとき、**ユーザーの Enter 入力 →
   1.5s 以内に持続出力（≥3 チャンク）が始まる**こと。サーバーは PTY の
   入力（ws → `entry.term.write`）と出力（`entry.term.onData`）を両方中継して
   いるので、そこをタップするだけ。新規依存なし。
2. **ユーザーボール（In Review）へ** = working 中に**出力静止 > 2.0s**
   （実測最大ギャップ 0.55s の 3.6 倍マージン）。完了も承認待ちも等しくここに落ちる。
3. **Enter 非先行の出力は working にしない。** 起動バースト・resume 時の画面再描画・
   アイドル中の孤立再描画（80B/数秒おき）は Enter が先行しないため構造的に無視される。
   起動直後のチラつき問題は閾値ではなく**このルールの形で消える**。
4. 定数は名前付きでサーバーに置く（例: `AGENT_SILENCE_MS = 2000`、
   `AGENT_OUTPUT_CONFIRM_MS = 1500`、`AGENT_MIN_CHUNKS = 3`）。チューニング可能に。

エージェント検出（分岐の判定方法）:

- `#{pane_current_command}` は npm インストールの claude だと `node` になり得るため
  単純一致は不可。**ペイン PID からプロセスツリーを辿り、コマンドラインに
  claude / codex をマッチさせる**（メモリ可視化 `server/process-memory.ts` の
  ps 機構を流用。実装時に実測で確認すること）。

セッション ID の紐づけ（シム廃止の代償と解決）:

- 用途は **L4 resume とカード自動命名だけ**に縮小（レーン移動は session ID 不要）。
- **transcript ファイルの発見で後から紐づける**: cwd スラグ
  （`~/.claude/projects/<slug>/`）配下で、エージェント起動直後に新規作成された
  `<sessionId>.jsonl` を相関（作成時刻）で結ぶ。既存の `/api/sessions?cwd=`
  セッション発見機構を流用。同一 cwd で複数カードが同時に claude を起動した場合の
  曖昧さは既知の制約として文書に明記（tie-break の工夫は Codex 裁量、過剰設計不要）。
- カード自動命名はセッション jsonl 内の `ai-title` 行を使える（実測で存在確認済み）。

- **論点2（カードとセッションの結びつき）を「決定済み」へ移動**: フックレス化により
  「hook を刺す仕組み」は不要になった。紐づけは transcript 発見方式。決定の記録
